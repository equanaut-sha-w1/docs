## Phase 1 -- Capture (Raw Session Notes)

Immediately after completing a browser workflow, record the following:

### 1.1 URL Journal

List every URL visited, in order. Include the navigation method.

| # | URL | Navigation Method | Page Title |
|---|-----|-------------------|------------|
| 1 | `http://127.0.0.1:5174/` | Direct navigation | E2S Pipeline Dashboard |
| 2 | `http://127.0.0.1:5174/` -> Priority tab | Click nav link | Priority Queue |
| 3 | `http://127.0.0.1:5174/` -> Actions tab | Click nav link | Actions |

### 1.2 Action Log

Record each action as a plain-language sentence:

- "Clicked the 'Priority' link in the top navigation bar"
- "Clicked 'Show all criteria' on Priority item #1"
- "Scrolled down to find the success row with View/Analyze/Export buttons"
- "Clicked 'Export All Runs' button in top-right of Dashboard"

### 1.3 Final State Screenshot

For each page visited, note the final observed state:
- What was visible on screen
- Any error messages or loading states
- Data values that confirm correct state

### 1.4 Prerequisites

Document everything that must be true before the workflow starts:

<Steps>
  <Step title="Backend API server running">
    API server must be running on port 8000
  </Step>
  <Step title="Frontend dev server running">
    Dev server must be running on port 5174
  </Step>
  <Step title="Pipeline status: connected">
    Green dot indicator showing backend connectivity
  </Step>
  <Step title="Processing history exists">
    At least one successful processing run in history (for View/Analyze/Export buttons)
  </Step>
  <Step title="Browser requirements">
    Any modern browser with JavaScript enabled
  </Step>
</Steps>

### 1.5 Flag Dynamic vs. Static Elements

| Element | Type | Why |
|---------|------|-----|
| Nav bar with 7 links | **Static** | Always present, never changes |
| Stat card values (0, 77.3%) | **Dynamic** | Changes with each poll cycle |
| Processing history rows | **Dynamic** | New rows appear, statuses change |
| Pipeline status dot (green/red) | **Dynamic** | Depends on backend connectivity |
| Top Priority card content | **Dynamic** | Changes after re-ranking |
| "not_started" badge | **Dynamic** | Reflects pipeline run state |
| System score (7.3/10) | **Semi-static** | Only changes when capability scorecard is regenerated |

## Phase 2 -- Structure (Step Block Format)

Transform raw notes into the standardized step-block format.

### Rules

1. **One action per step** -- never combine "click X and then type Y" into one step
2. **Imperative verb title** -- "Navigate to," "Click," "Verify," "Scroll to," "Enter," "Select"
3. **Observable expected result** -- what the operator can see, not internal state
4. **Verification method** -- how to confirm the step succeeded

### Step Block Template

```markdown
### Step N: [Verb] [Object]
- **URL:** `[current URL]`
- **Action:** [Single atomic action]
- **Selector/Target:** `[how to find the element]`
- **Input (if any):** `[value]`
- **Expected Result:** [What visibly changes]
- **Verification:** [How to confirm]
- **Pre-requisite:** [What must be true first]
```

## Phase 3 -- Validate

Run these checks against your completed guide:

### 3.1 Read-Through Test

- [ ] Read each step aloud. Can someone unfamiliar with the app follow it?
- [ ] Does every step have exactly one action?
- [ ] Are all imperative verbs concrete? (Click, Navigate, Scroll, Verify, Enter -- not "Check" or "Do")

### 3.2 Selector Audit

- [ ] Every selector/target description is specific enough to find the element
- [ ] No reliance on array indices that shift (e.g., "3rd row" without stable identifier)
- [ ] Dynamic content flagged with warnings

### 3.3 Dependency Check

- [ ] Steps are in correct order -- no step references a result from a later step
- [ ] Prerequisites are listed at the top, not buried in step blocks
- [ ] Each step's pre-requisite field correctly references the prior step

### 3.4 Dry-Run on Fresh Profile

- [ ] Open incognito/fresh browser profile
- [ ] Follow every step exactly as written
- [ ] Record any step that fails or is ambiguous
- [ ] Update the guide with corrections
